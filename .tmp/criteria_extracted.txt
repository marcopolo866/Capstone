Project Title: LLM-Engineered Graph Algorithms: Can ChatGPT Beat the Classics? 
Client: Hasan Jamil 
 
Project Summary: 
 
Students will rigorously test whether ChatGPT can generate efficient C++ implementations for 
three problems: 
1. Shortest path between two nodes (weighted graphs) 
2. Shortest path between two nodes via a specified intermediate node 
3. Subgraph isomorphism (deciding/locating if a pattern graph occurs in a target graph) 
The workflow is: (a) prompt ChatGPT to write C++20 code, (b) compile, repair, and minimally 
refactor to correctness, (c) optimize within strict rules, and (d) benchmark against strong 
baselines. Students must document all remediation done to make LLM code correct and fast, 
and then compare performance and asymptotic behavior to leading algorithms. The stretch goal 
is to outperform robust baselines on at least some datasets; the required goal is a transparent, 
reproducible evaluation of how close LLM-produced code can get and what engineering it takes. 
 
Rules & Scope: 
 
• 
Language & toolchain: C++20, CMake, GCC/Clang, no high-level graph libraries (e.g., 
no Boost Graph for core algorithms). STL, <ranges>, <queue>, <vector>, 
<unordered_*>> allowed. 
• 
LLM constraint: Core algorithmic code must originate from ChatGPT prompts (students 
may prompt iteratively). Students can fix correctness/perf bugs, change data structures, 
and tune constants – but no wholesale rewrites from scratch without showing LLM 
provenance. Every ChatGPT response must be analyzed and reflected upon and logged 
– Cognitive engagement. 
• 
Baselines (reference implementations): 
o Shortest path: Dijkstra (binary heap / pairing heap) and Dial’s (for small integer 
weights). Optional: A* for spatial graphs. 
o Via-node path: Decompose into two shortest paths (s→v + v→t) with shared 
preprocessing when applicable (e.g., multi-source runs). 
o Subgraph isomorphism: state-of-the-art open approaches include VF2/VF3, 
Glasgow/RI/LAD families. Use at least two strong baselines (open-source or well-
cited academic implementations). 
• 
Datasets: Include both synthetic (Erdős–Rényi, Barabási–Albert, grid/road-like) and real 
graphs (e.g., SNAP road networks, DIMACS shortest path, PACE/VF2 benchmark sets 
for subgraph iso). 
• 
Metrics: wall-clock time, memory, node/edge visits, branch/backtrack counts, and 
scalability curves vs |V|, |E|, and pattern size (for iso). 
• 
Fairness: Same machine, compiler flags, and time/memory limits; 3–5 independent runs 
with medians & error bars. 
• 
Reproducibility: Deterministic builds, version-pinned toolchain, scripts for data 
acquisition, build, and run. 
 
Required Components: 
 
1. Prompting Protocol & LLM Provenance 


===== PAGE 2 =====
a. A written, versioned prompt protocol (seed prompts, follow-ups, rejection 
sampling strategy, temperature settings). 
b. Archive of all prompts/responses, tagged to final code modules. 
c. Rationale for prompt wording and observed LLM behavior (e.g., typical errors, 
hallucinated APIs). 
2. C++ Implementations (LLM-Originated + Minimal Remediation) 
a. SP-S2S: Single-pair shortest path on weighted graphs. 
b. SP-Via: s→v→t shortest path with one designated intermediate node; evaluate 
both naïve (two Dijkstra runs) and LLM-invented variants (e.g., bidirectional, 
landmarks, potentials) if produced. 
c. SUBISO: Subgraph isomorphism (decision + one witness mapping). 
d. For each: clearly separate LLM-authored core from student remediations (diffs, 
comments). 
3. Correctness Harness & Oracles 
a. Unit tests (GoogleTest or Catch2) with small hand-crafted cases (including edge 
cases: negative-like traps, disconnected graphs, multi-edges, high-degree hubs). 
b. Cross-check vs baseline outputs (exact path length, mapping validity). 
c. Fuzz tests for graph generators; property-based checks (e.g., path optimality 
inequality). 
4. Performance Engineering & Instrumentation 
a. Unified benchmark harness (Google Benchmark or custom) with CLI to select 
datasets, seeds, and time limits. 
b. Profiling (perf/VTune/Callgrind), sanitizers (ASan/UBSan), and -O3 -
march=native builds. 
c. Micro-optimizations documented (e.g., custom allocators, adjacency layout, 
small-vector optimization, cache-aware queues) and their measured impact. 
5. Baselines & Comparative Evaluation 
a. Build and run at least two strong baselines per task. 
b. Report speedup (or slowdown), memory, and scalability vs graph 
size/density/pattern size. 
c. For subgraph iso, include pruning heuristics analysis (degree/orderings, 
refinement, domain filtering) and backtrack statistics. 
6. “Remediation Ledger” (LLM Cleanup Accounting) 
a. A structured log quantifying fixes: 
i. Compile-time: missing headers, wrong types, API mismatches. 
ii. Runtime: incorrect results, off-by-one, overflow, UB. 
iii. Algorithmic: wrong complexity, non-terminating search, missing pruning, 
data-structure choices. 
b. For each fix: lines changed, time spent, why needed, impact on correctness/perf. 
c. Summaries: total edits, percent of LLM code retained, performance before/after 
each major fix. 
7. Final Report (Scholarly Style) 
a. Problem statements & complexity expectations. 
b. Methodology: prompting protocol, datasets, baselines, hardware, flags. 
c. Results: tables/plots (with CIs), ablations (which edit helped most), failure cases. 
d. Analysis: where LLM shines vs fails (e.g., boilerplate vs deep pruning logic), 
generalization across graphs, reproducibility. 
e. Conclusion: whether and when LLMs can author competitive graph algorithms; 
recommendations for future workflows (e.g., human-in-the-loop optimization). 
8. Artifacts for Reproducibility

a. Public repo (or course archive) with: code, prompts, diffs, scripts, dataset links, 
Dockerfile/Conda env, CI config. 
b. “One-command” script to rebuild and rerun key experiments. 
 
Success Criteria (Tiered): 
• 
Pass: Correct algorithms, clean benchmarks, rigorous comparison, full remediation 
ledger. 
• 
Merit: Match baseline performance (±20%) on ≥50% of cases; win on at least a niche 
family (e.g., sparse road networks or small-pattern iso). 
• 
Distinction (Stretch Goal): Beat at least one strong baseline consistently on a predefined 
subset, with credible analysis explaining why (e.g., cache-friendly layout, domain-
specific ordering, superior pruning learned via iteration). 
Note: Beating the very best subgraph iso solvers universally is unlikely; you’ll be graded on rigor 
and insight, not just raw wins. 
 
Deliverables Checklist: 
1. Design Doc: problem scope, baselines, datasets, prompting protocol. 
2. Codebase: LLM-origin files + remediation diffs, CMake project, tests, benchmarks. 
3. Remediation Ledger: structured CSV/markdown of all fixes with impact. 
4. Benchmark Package: scripts + configs to reproduce all tables/figures. 
5. Final Report & Slides: results, analysis, and lessons about LLM sophistication. 
 
Learning Outcomes: 
By completing this project, students will be able to: 
1. Apply advanced graph algorithm concepts 
a. Demonstrate understanding of classic shortest path algorithms (e.g., Dijkstra, 
Dial’s, bidirectional search) and subgraph isomorphism methods (e.g., VF2, LAD, 
Glasgow). 
b. Analyze algorithmic complexity and scalability for different graph structures and 
problem variants. 
2. Leverage and evaluate large language models (LLMs) in software engineering 
a. Formulate effective prompts to guide ChatGPT in producing C++ 
implementations of non-trivial algorithms. 
b. Assess the correctness, completeness, and efficiency of LLM-generated code 
against formal specifications. 
3. Integrate compiler, debugging, and optimization skills 
a. Diagnose and remediate compile-time and runtime errors in third-party (LLM-
authored) code. 
b. Apply micro-optimizations (data structure choice, memory layout, algorithmic 
tweaks) to improve performance. 
4. Design and execute reproducible performance experiments 
a. Select representative synthetic and real-world graph datasets. 
b. Benchmark algorithm performance using rigorous, repeatable methodology 
(multiple runs, hardware control, statistical reporting). 
c. Compare LLM-derived algorithms against state-of-the-art baselines in both 
correctness and speed. 
5. Practice empirical software analysis 
a. Quantify the engineering effort required to transform LLM-generated code into 
working, efficient solutions. 
b. Maintain a “remediation ledger” documenting changes, rationale, and impact on 
performance and correctness.

6. Communicate technical results effectively 
a. Produce a scholarly-style technical report detailing methodology, datasets, 
experiments, results, and analysis. 
b. Present findings orally and visually, including live demonstrations and 
comparative performance plots. 
7. Critically evaluate AI-assisted programming workflows 
a. Identify where LLMs succeed (e.g., boilerplate, known algorithm templates) and 
where they fail (e.g., deep optimization, pruning strategies). 
b. Articulate guidelines for effectively integrating LLMs into performance-critical 
software development.
