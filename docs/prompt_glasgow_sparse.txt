OVERVIEW
--------
You are tasked with implementing a high-performance subgraph matching solver in
C++. This prompt is for the Glasgow-family `.lad` branch of the Capstone
subgraph benchmark.

Your implementation must solve exactly the same mathematical problem that the
current benchmark pipeline evaluates:

  Count ALL injective embeddings of a simple undirected vertex-labelled pattern
  graph H into a simple undirected vertex-labelled target graph G under
  non-induced semantics.

Do NOT implement or reference any named algorithm (for example VF2, VF3,
Glasgow, Ullmann, RI, LAD, etc.). Derive your own approach from first
principles. Your solution will be evaluated on runtime speed and memory usage.


PROBLEM DEFINITION
------------------
Given a pattern graph H and a target graph G, both simple, undirected, and
vertex-labelled, count all injective mappings f: V(H) -> V(G) such that:

  (1) EDGE PRESERVATION (undirected):
      For every undirected edge {u, v} in H, the undirected edge
      {f(u), f(v)} must exist in G.

  (2) NON-INDUCED SEMANTICS:
      Extra edges between mapped target vertices are allowed.
      In other words, if {f(u), f(v)} exists in G while {u, v} does not
      exist in H, that does NOT invalidate the mapping.

  (3) INJECTIVITY:
      f must be injective. No two distinct pattern vertices may map to the
      same target vertex.

  (4) VERTEX LABEL MATCHING:
      Every vertex carries an integer label, and labels must match exactly:
        label(u) == label(f(u))
      There are no edge labels.

  (5) ENUMERATION:
      Count ALL valid mappings. Do not stop at the first match.
      Do not apply symmetry breaking. Automorphic variants count separately.

  (6) GRAPH CLASS:
      Graphs are simple and undirected:
      - no self-loops
      - no parallel edges
      - adjacency is symmetric
      Pattern graphs may be disconnected and may contain isolated vertices.


INPUT FORMAT
------------
Your program must accept two file paths as command line arguments:

  ./solver <pattern_file> <target_file>

Use exactly this interface:

  - `argv[1]` is the pattern graph path
  - `argv[2]` is the target graph path
  - expect exactly two non-option positional arguments
  - do not require stdin, prompts, config files, environment variables, or
    extra command line flags in the benchmark path

The benchmark uses the current Glasgow-family vertex-labelled LAD text format
(stored as `.lad`) shown below:

  Line 1:         number of vertices n
  Next n lines:   <vertex_label> <degree> <neighbor_0> <neighbor_1> ...

Notes:
  - Vertex ids are implicit: the first adjacency row is vertex 0, the second is
    vertex 1, and so on.
  - Graphs are undirected, so each undirected edge appears in both endpoint
    rows in benchmark-generated inputs.
  - The active generated Glasgow inputs use the `vertexlabelledlad` flavor:
    each row begins with the vertex label, then the degree, then neighbour ids.
  - There are no edge labels.

Example (3-vertex path 0-1-2, all labels = 1):

  3
  1 1 1
  1 2 0 2
  1 1 1


OUTPUT FORMAT
-------------
Print exactly one parseable integer solution count to stdout:

  <count>

where `<count>` is the total number of valid injective non-induced embeddings.

Do not rely on any external files or sidecar outputs. Print only the count in
the normal success path. Do not print explanations, labels, timing lines,
debugging logs, Markdown, or any extra prose to stdout.


PERFORMANCE REQUIREMENTS
------------------------
Your implementation will be benchmarked mainly on sparse, structured graph
instances:

  - Pattern graph H: sparse, low average degree, informative local structure
  - Target graph G: sparse, with discriminative neighbourhood shapes
  - Vertex labels and local topology together provide strong pruning signal

In this regime, neighbour relationships are highly informative. Knowing that a
pattern vertex maps to a target vertex should strongly restrict the feasible
placements of adjacent pattern vertices.

Optimize aggressively for:

  (a) RUNTIME SPEED
      - Exploit local structure and label constraints as early as possible
      - Use a strong pattern-vertex ordering based on structural
        discriminativeness
      - Prune candidate mappings immediately when neighbourhood consistency
        fails
      - Avoid redundant work on states that are effectively equivalent

  (b) MEMORY EFFICIENCY
      - Minimise heap allocations in the search loop
      - Keep search state compact
      - Do not materialize every mapping if only the count is required
      - Prefer reused buffers or compact stack state over repeated allocation

Standard C++20 and the STL are permitted. No external libraries.
